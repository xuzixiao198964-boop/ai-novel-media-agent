# 测试说明（AI 小说生成 Agent 系统）

## 服务器环境约束

| 资源 | 实际值 | 测试影响 |
|------|--------|----------|
| CPU | 2核 | 并发测试受限 |
| 内存 | 1GB（可用~476MB） | 禁止大内存测试 |
| 磁盘 | 20GB（仅剩~1GB） | 测试数据必须及时清理 |
| 部署 | 裸机 systemd（非Docker） | 直接在系统环境运行 |
| 数据库 | PostgreSQL 16（唯一存储） | 无Redis，缓存用进程内 |
| 任务队列 | PostgreSQL任务表（无RabbitMQ） | 无MQ相关测试 |

**测试约束：**
1. 并发测试最多2个并行任务（内存限制）
2. 测试前后检查磁盘空间，测试数据及时清理
3. 数据库连接池不超过5个连接
4. 单任务内存峰值不超过300MB
5. 所有缓存测试使用PostgreSQL或内存mock，无Redis
6. 系统测试不依赖 Docker；服务以 **systemd** 管理

## 系统测试策略（验收 / 与架构对齐）

**定义**：系统测试在**真实部署形态**下验证端到端行为——**单进程模块化单体**（FastAPI）、**PostgreSQL 16**、本地目录存文件、**无 Redis / RabbitMQ / Docker**。

| 维度 | 要求 |
|------|------|
| 执行环境 | 优先 `104.244.90.202` 裸机；或资源等价的验收机（2核/1GB/约20GB盘） |
| 前置检查 | 磁盘剩余 **≥500MB**（否则禁止新建生成类任务）；`free -h` 确认可用内存 |
| 服务管理 | `systemctl` 启停与重启；日志用 `journalctl -u <unit> -n 200` |
| 数据 | 使用测试账号与可清理输出目录；测毕执行清理避免盘满 |
| 与详细设计对应 | 幂等：`idempotent_keys` 表；队列：`task_queue` 表；缓存：进程内或 `cache_entries` |

**用例分层**：

| 层级 | 编号 | 说明 |
|------|------|------|
| 部署冒烟 | TC-S01、TC-S02 | systemd 与日志，每次发版必跑 |
| 资源门禁 | TC-C11、TC-C12 | 磁盘与内存，P0/P1 |
| 业务 E2E | TC-C01–TC-C10 | 商业化关键路径，见下文 |

**与单元测试文档关系**：细粒度用例与 Mock 约定见 `docs/requirements/testing_documentation.md`（§0、§6.3）；本文档侧重 **可手工或可脚本化** 的系统级步骤。

### TC-C02: 用户充值→余额扣减→余额不足暂停→再充值→恢复

| 项目 | 内容 |
|------|------|
| **测试ID** | TC-C02 |
| **描述** | 验证余额模式下的充值、扣减、暂停和恢复流程 |
| **优先级** | P0 |
| **前置条件** | 已注册用户；支付沙箱环境可用 |

**步骤：**
1. 用户充值 50 元到余额账户
2. 发起小说生成任务，系统按 成本×1.15 扣减余额
3. 查询余额，确认扣减金额正确
4. 再次发起任务，余额不足时任务自动暂停
5. 用户再次充值 100 元
6. 暂停任务自动恢复继续执行

**预期结果：**
- 充值后余额 = 50.00 元
- 扣减后余额 = 50.00 - (实际成本 × 1.15)，精确到分
- 余额不足时任务状态为 `paused`，错误原因为 `insufficient_balance`
- 再次充值后余额 = 剩余 + 100.00
- 暂停任务自动恢复，状态变为 `running`

| 项目 | 内容 |
|------|------|
| **测试ID** | TC-C07 |
| **描述** | 验证套餐升级时权益即时切换和配额折算逻辑 |
| **优先级** | P1 |
| **前置条件** | 用户持有基础套餐活跃订阅，已使用部分配额 |

**步骤：**
1. 用户当前为基础套餐（月付），本月已用 3/10 小说配额
2. 用户选择升级到专业套餐
3. 系统计算剩余基础套餐费用，折抵升级差价
4. 用户补差价支付成功
5. 检查升级后的套餐类型和权益
6. 检查配额是否正确折算（基础剩余7次 → 按比例转为专业套餐配额）

**预期结果：**
- 升级补差价 = PRO月费 - BASIC月费 × (剩余天数/总天数)
- 支付成功后，套餐立即变为 PRO
- 新权益立即生效（更高的配额上限、优先队列等）
- 旧配额按比例折算：新剩余 = 旧剩余 × (PRO配额 / BASIC配额)
- 升级记录包含 `upgrade_from`、`prorated_credit` 字段

### TC-C08: 支付回调幂等性（重复回调不重复入账）

| 项目 | 内容 |
|------|------|
| **测试ID** | TC-C08 |
| **描述** | 验证支付网关重复发送回调时系统不会重复处理 |
| **优先级** | P0 |
| **前置条件** | 已创建待支付订单 |
| **实现说明** | 幂等键使用 PostgreSQL `idempotent_keys`（或等价唯一约束），**非 Redis SETNX** |

**步骤：**
1. 创建支付订单（支付宝渠道）
2. 模拟支付宝发送第一次支付成功回调
3. 验证订单状态变为 PAID，用户权益激活
4. 模拟支付宝发送第二次相同回调（重试）
5. 再模拟第三次相同回调
6. 检查订单状态和用户余额/权益

**预期结果：**
- 第一次回调：订单 PENDING → PAID，权益激活
- 第二次回调：系统返回 `success`（告知网关已处理），但不重复入账
- 第三次回调：同上，幂等处理
- 订单表中只有一条支付记录
- 用户余额/配额只增加一次
- 所有回调均返回 `success` 给支付网关（避免网关持续重试）

### TC-C10: 每日对账（订单vs支付网关记录匹配）

| 项目 | 内容 |
|------|------|
| **测试ID** | TC-C10 |
| **描述** | 验证每日自动对账机制的准确性和异常检测能力 |
| **优先级** | P1 |
| **前置条件** | 前一日有若干笔已完成支付订单；支付网关对账文件可获取 |

**步骤：**
1. 触发每日对账任务（`POST /admin/reconcile?date=昨日`）
2. 系统拉取支付宝、微信、抖音各渠道的昨日对账文件
3. 逐笔与本地订单记录进行匹配
4. 插入一笔故意不匹配的记录（网关有、本地无）

**预期结果：**
- 正常订单全部匹配（`matched` 数量正确）
- 检测到不匹配记录（`missing_in_local` ≥ 1）
- 对账报告包含：匹配数、不匹配数、本地缺失数、网关缺失数
- 不匹配记录包含详细信息（订单号、金额、渠道、差异原因）
- 对账完成后自动发送告警通知（如有不匹配）
